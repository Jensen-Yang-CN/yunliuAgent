## 二、架构设计类（重点准备）

### ★ 3. 为什么设计在线/离线双模架构？

这个项目实际部署在工地环境中，网络条件不稳定。如果完全依赖云端 LLM，一旦断网整个智能体就不可用了。所以设计了"在线优先、离线兜底"的双模架构：

- **在线模式**：调用千问大模型做意图理解，能处理复杂的自然语言指令，生成高质量的分析报告；
- **离线模式**：通过关键词匹配识别用户意图（比如"安全帽""检测"触发 helmet_detection），直接调用工具并用模板生成回复；
- **自动降级**：在线模式发起的 try 块如果遇到网络超时、API 异常，会在 except 中自动切到离线模式，用户无感知。

这样既保证了智能化上限，又保证了基础功能的可用性下限。

---

### ★ 4. 为什么选择 Function Calling 这套模式？有什么好处？

Function Calling 是 OpenAI 定义的一套工具调用协议，本质上是把 Python 函数的签名、参数类型和用途描述，翻译成 LLM 能理解的标准 JSON Schema，好处是：

- **标准化**：任何兼容 OpenAI 协议的 LLM（千问、GPT 等）都能理解并返回结构化的调用指令，不绑定特定厂商；
- **扩展性**：项目里封装了安全帽检测和 VLM 视频分析两个工具，未来要加新能力（比如火焰检测、人员计数），只需注册一套新的 schema + 实现函数，对话流程完全不用改；
- **LLM 不直接执行代码**：LLM 只返回调用指令（`tool_calls`），由我的 agent 代码去实际执行，再把结果回填给 LLM 做最终总结——这个"Schema 定义 → LLM 返回指令 → 代码执行回填"的三步闭环，是 Agent 工具调用最核心的范式。

**以安全帽检测为例，发给 LLM 的工具描述长这样**（来自 [`tools.py`](../MultiDetection-backend/app/tools.py#L211-L224)）：

```json
{
  "type": "function",
  "function": {
    "name": "detect_helmet",
    "description": "检测视频或图片中的安全帽和人员。未提供file_path时自动从默认目录选第一个MP4。",
    "parameters": {
      "type": "object",
      "properties": {
        "file_path": {
          "type": "string",
          "description": "要检测的视频或图片文件路径"
        },
        "confidence_threshold": {
          "type": "number",
          "description": "检测置信度阈值，0-1，默认0.5"
        }
      }
    }
  }
}
```

**LLM 收到后会返回如下调用指令**：

```json
{
  "tool_calls": [{
    "function": {
      "name": "detect_helmet",
      "arguments": "{\"file_path\": \"/path/to/video.mp4\", \"confidence_threshold\": 0.6}"
    }
  }]
}
```

然后我的 agent.py 解析这个指令 → 调用 `helmet_detection_tool.detect_from_file()` → 把返回结果回填给 LLM → LLM 生成自然语言总结。整套流程 LLM 全程不动手执行，只做"决策大脑"。

---

### 5. 决策轨迹（思维链可视化）的设计思路是什么？为什么不用 LLM 原生的 thinking 输出？

有两个考虑：

- **可控性**：LLM 的 thinking 输出不可控，格式不稳定，可能暴露不该给用户看的内部思考；
- **通用性**：离线模式没有 LLM，但也需要展示决策过程。

所以我设计了一套标准化的四步轨迹——**用户输入 → 模型决策 → 工具执行 → 最终回复**——无论在线还是离线都按这个结构输出，前端用 `el-timeline` 时间线统一渲染，四条轨迹分别用不同颜色区分。

本质上是把 "Chain of Thought" 从模型内部搬到了系统中台，变成**可观测、可审计的标准化推理链路**。

## 三、技术实现类

### 6. 前端 AgentChat.vue 的数据流是怎样的？

用户输入 → POST `/api/agent/chat` → 后端返回统一 JSON 结构（`message` + `tool_calls` + `tool_results` + `decision_trace`）→ Vue 将每条消息 push 到 messages 数组 → 模板中 `v-for` 渲染：

- 决策轨迹用 Element Plus 的 `el-timeline` 时间线组件展示，按类型着色；
- 工具调用和结果用 `el-collapse` 折叠面板展示 JSON；
- 如果有 `preview_stream_url` 就渲染 `<img>` 标签拉取 MJPEG 流；
- 预留了 WebSocket 通道用于流式返回，当前默认走 HTTP。

---

### 7. MJPEG 实时预览流是怎么内嵌在对话页里的？

后端在安全帽检测完成后，进程内将视频文件路径绑定到 `video.py` 中预定义的 7 号通道，生成流地址 `/api/video/video_stream/7` 返回给前端。前端检测到工具结果中有 `preview_stream_url` 字段时，直接在对话气泡里渲染一个 `<img>` 标签，src 指向这个流地址。浏览器会把 MJPEG 流当作持续加载的图片逐帧显示，不需要 WebSocket 或 WebRTC。

---

### ★ 8. 遇到什么技术难点？怎么解决的？

一个比较典型的问题是**自动降级时的状态一致性**：在线模式走到一半失败了，切换离线模式时不能让前端拿到半截数据。我的处理是在 `agent.py` 的 chat 接口里用 try-except 包裹整个在线调用，异常时**完全重新走一遍离线流程**——从意图识别到工具执行到回复生成——保证返回给前端的数据结构始终完整。

另外离线模式的模板回复太生硬，我后续优化了模板拼接逻辑，把工具结果的摘要数据（人数、安全帽数等）动态填入模板，让离线回复也有基本的可读性。

## 四、场景追问类

### 9. 如果让你重新设计这个系统，会做哪些改进？

三个方向：

- **流式输出**：当前走 HTTP 一次返回全部结果，用户等待时间较长。可以接入 SSE 或 WebSocket 做流式，把决策轨迹逐条推送，用户感知更实时；
- **工具编排能力**：目前工具之间是独立调用的。如果要做"先检测安全帽，发现违规后再自动截取相关帧做 VLM 分析"这种串联，需要引入 Tool Chaining / Orchestration，让 LLM 能连续多轮调用不同工具；
- **工具扩展机制**：目前的工具注册还是硬编码在代码里的，可以改成配置文件或插件化注册，进一步降低加新工具的接入成本。

---

### 10. 这个系统的安全性怎么考虑的？

因为本项目是内部工地使用，网络相对封闭，没有做复杂的鉴权层。但有几个基础考量：API Key 走环境变量不硬编码、前端请求走 Vite 代理避免跨域暴露后端端口、对话历史可清除避免敏感信息残留。如果是面向公网发布，还需要加用户认证和 API 限流。

---

### 11. 为什么用千问而不是 GPT？

千问的 DashScope API 兼容 OpenAI 格式，接入成本低；国内网络环境直连千问比连 OpenAI 稳定；API 费用也更可控。从技术角度，换用 GPT 或其他兼容模型只需要改 `base_url` 和 `api_key`，代码层面是零改动。

## 七、面试一句话总答复（智能体整体流程）

> "整个智能体系统的核心链路是：前端 AgentChat 发送自然语言 → agent.py 根据环境变量选择在线（千问）或离线（关键词）模式 → 在线模式下 llm_agent.py 将工具 schema 发给千问，千问返回 tool_calls 指令后执行对应的 Python 函数，结果回填再生成总结；离线模式下直接关键词匹配调用工具，模板拼回复。两种模式返回统一结构的 JSON（message + tool_calls + tool_results + decision_trace），前端用 el-timeline 时间线组件渲染四步决策轨迹。"

**三个后端文件分工：**

| 文件 | 一句话定位 |
|------|-----------|
| `agent.py` | 路由分发 + 双模切换（"用在线还是离线"） |
| `llm_agent.py` | 在线大脑（"怎么跟千问对话，怎么把 tool_calls 变成实际调用"） |
| `tools.py` | 工具实现（"检测怎么跑，VLM 怎么调"） |
