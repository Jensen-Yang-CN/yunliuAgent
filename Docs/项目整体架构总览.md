## 3. 核心能力与数据流

### 3.1 多通道目标检测（8 路）
- 上传：
  - `POST /api/video/upload/{channel_id}`（0~7）
  - 保存到 `app/static/uploads/channel_{id}_xxx.mp4`
- 取流（MJPEG）：
  - `<img src="/api/video/video_stream/{channel_id}">`
  - 服务端生成器循环：OpenCV 读取帧 → YOLO 推理 → 绘制 → JPEG 编码 → 持续推送
- 状态：
  - `video.py` 维护每路的 `video_paths/current_frames/is_video_running/统计` 等数组
  - 通道数由 `CHANNELS = 8` 统一控制

页面：`VideoGrid.vue`
- 每个单元格 = 一个通道：上传→成功后 `<img>` 直连 MJPEG 流
- 已从 4 路扩展为 8 路（v-for 和本地状态已更新）

### 3.2 AI 智能体对话（工具调用 + 决策轨迹 + 实时预览）
- 入口：`POST /api/agent/chat`（`agent.py`）
- 工具（`tools.py`）：
  1) `detect_helmet(file_path?, confidence_threshold=0.5)`
     - 未传 `file_path`：自动从 `MultiDetection-backend/output_parts` 选择第一个 MP4
     - 返回检测摘要与统计
  2) `vlm_start_video_analysis()` / `vlm_stop_video_analysis()`
     - 调用 YunliuAIVideo `16532` 的 `/start_video_analysis` 与 `/stop_video_analysis`
- 离线关键词兜底（保证可用）：
  - 包含“安全帽/检测/helmet”→ 直接调用 `detect_helmet`
  - 包含“启动 大模型视频/视频分析/VLM”→ 调 `vlm_start_video_analysis`
  - 包含“停止 大模型视频/视频分析/VLM”→ 调 `vlm_stop_video_analysis`
- 决策轨迹：
  - user_message → llm_decision（或兜底决策）→ tool_execution → final_response
- 实时预览（仅“安全帽检测”）：
  - 智能体在检测完成后，**进程内**把通道 7 绑定到 `used_file`
  - 前端 `AgentChat.vue` 自动识别 `preview_stream_url`（`/api/video/video_stream/7`）并显示为逐帧 MJPEG

页面：`AgentChat.vue`
- 支持 HTTP 回退发送
- “查看工具”已做友好映射：
  - 安全帽检测（detect_helmet）
  - 大模型视频分析（整合 vlm 启停）
- 生成的内容区、工具结果区、预览区均支持滚动；增加“显示/隐藏实时预览”按钮

### 3.3 大模型视频分析（VLM）
- 服务：`YunliuAIVideo`（`server.py` 聚合）
  - `POST /start_video_analysis`：启动视频分析
  - `POST /stop_video_analysis`：停止视频分析
- 控制方式：
  - 在 `AgentChat.vue` 输入“启动大模型视频推理/停止大模型视频推理”，智能体调用对应工具触发
  - 原生可视化页：`VideoAnalysis.vue`

## 6. 常见问题（Troubleshooting）

- 前端不滚动：
  - Vite 模板的 `body { display: flex; place-items:center; }` 会锁死视图
  - 已改为 `display:block; overflow-y:auto;`（见 `src/style.css`）

- MJPEG 预览不显示：
  - 检查工具结果 JSON 是否含 `preview_stream_url`
  - 直接在新标签打开该 URL 是否显示流
  - 后端 `video.py` 的通道 7 是否已绑定、`is_video_running[7]` 是否被重置

- H.264 写视频失败（YunliuAIVideo）：
  - `openh264-1.8.0-win64.dll` 缺失或版本不匹配
  - 解决：要么安装对应 DLL，要么将 `cv2.VideoWriter_fourcc` 改为 `mp4v/MJPG` 并匹配后缀

- 千问工具调用 400（函数参数结构）：
  - `function.arguments` 必须是字符串；assistant.tool_calls 必须为 OpenAI 兼容结构
  - 当前使用离线兜底策略，仍能保证功能可用

## 9. 术语对照（Glossary）

- MJPEG：基于 multipart/x-mixed-replace 的 JPEG 序列流（<img> 可直接渲染）
- 通道（Channel）：多路视频的编号（0~7）
- 工具（Tool）：LLM 的函数调用能力中对外暴露的可执行功能
- 决策轨迹（Decision Trace）：对话中可见的“观察到的步骤”，不泄露模型内部思维链
