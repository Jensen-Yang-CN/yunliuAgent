# “功能”与“工具”的区别及设计说明

本文档旨在澄清“功能函数”与大语言模型（LLM）可调用的“工具”之间的核心区别，并详细说明如何将一个已有的Python功能“包装”成能被智能体（如千问）理解和调度的工具。

---

## 1. 核心区别：“功能” vs “工具”

### 功能 (Function)
- **是什么**：是你编写的一段具体的、可执行的代码，比如 `detect_from_file()`。
- **谁能用**：只能由开发者在代码中显式调用。
- **它懂什么**：只理解代码层面的输入（如函数参数）和输出（返回值）。它不理解“人话”。
- **例子**：`detect_from_file(file_path="video.mp4", confidence_threshold=0.6)`

### 工具 (Tool)
- **是什么**：是**你的功能函数 + 一份给LLM看的“说明书”**。
- **谁能用**：大语言模型（LLM）。LLM通过阅读“说明书”来决定是否以及如何使用它。
- **它懂什么**：LLM通过“说明书”中的自然语言描述来理解工具的用途，从而将用户的“人话”指令映射到具体的代码执行上。
- **例子**：你对智能体说“帮我看看有没有人没戴安全帽”，LLM阅读所有工具的说明书后，发现`detect_helmet`这个工具最匹配，于是决定调用它。

**一句话总结：**
> **工具 = 代码功能 + LLM能看懂的说明书 (JSON Schema)**

---

## 2. 为什么要使用“工具”模式？

将功能包装成工具，是为了**赋予大语言模型（LLM）与真实世界交互的能力**。

1.  **扩展LLM的能力**：LLM本身是一个“虚拟大脑”，它不会看视频、不会查数据库、不会控制你的电脑。通过工具，它能“指挥”你的代码去完成这些它本身做不到的事情。
2.  **连接虚拟与现实**：工具是LLM（虚拟）与你本地的文件系统、硬件、外部API（现实）之间的桥梁。
3.  **实现真正的自然语言接口**：你不再需要为每个功能都设计一个按钮或复杂的界面。用户只需要用自然语言下达指令，智能体就能自动分析并执行正确的操作。这大大提升了系统的智能化和易用性。

---

## 3. 如何设计一个LLM工具？（以“安全帽检测”为例）

将你的 `detect_from_file()` 功能包装成 `detect_helmet` 工具，需要以下两步：

### 第一步：找到核心功能函数
这很简单，就是你已经写好的 `detect_from_file()`。

### 第二步：为这个功能写一份“说明书”
这份“说明书”就是一个遵循特定格式的JSON对象，我们称之为**工具定义 (Tool Definition)**。它包含三个核心部分：

1.  **`name` (名称)**
    *   给工具起一个简洁、有意义的英文名，比如 `detect_helmet`。这是LLM调用时的唯一标识。

2.  **`description` (描述)**
    *   **这是最关键的一步**。用清晰、准确的自然语言描述这个工具能做什么、在什么场景下使用。LLM会根据这个描述来决定是否调用它。
    *   **好的描述示例**：“检测视频或图片中的安全帽和人员。当用户想检查安全合规、寻找未戴安全帽的人时使用。如果用户没有提供具体的文件路径，这个工具会自动检测一个默认的视频。”

3.  **`parameters` (参数)**
    *   定义这个工具需要哪些输入信息。每个参数都应该有自己的类型（如`string`, `number`）和描述。
    *   **`file_path`**：类型是`string`，描述是“要检测的视频或图片文件的路径”。我们可以不把它设为必需（`required`），这样LLM就知道在用户没提具体文件时可以不提供这个参数。
    *   **`confidence_threshold`**：类型是`number`，描述是“检测的置信度阈值”，可以给一个默认值。

---

## 4. 工具的格式要求 (OpenAI/千问兼容格式)

工具的“说明书”必须遵循一个标准的JSON结构。下面是 `detect_helmet` 工具的完整定义，位于 `app/tools.py` 中：

```json
{
  "type": "function",
  "function": {
    "name": "detect_helmet",
    "description": "检测视频或图片中的安全帽和人员。若未提供file_path，将自动从默认目录中选择第一个MP4文件进行检测...",
    "parameters": {
      "type": "object",
      "properties": {
        "file_path": {
          "type": "string",
          "description": "要检测的视频或图片文件的路径..."
        },
        "confidence_threshold": {
          "type": "number",
          "description": "检测置信度阈值，0-1，默认0.5"
        }
      }
    }
  }
}
```

## 5. 智能体如何调用工具（运行流程）

整个过程就像一个团队协作：

1.  **你 (用户)**：下达指令：“帮我检测一下工地的安全帽佩戴情况”。
2.  **智能体 (`agent.py`)**：作为项目经理，接收到你的指令。如果是在线模式，它会将你的话和它手上的所有工具的“说明书”一起打包，发给**千问LLM**。
3.  **千问LLM (大脑)**：阅读所有说明书，发现 `detect_helmet` 工具的描述最符合你的要求。于是它做出决策：“好的，我要调用 `detect_helmet` 这个工具。因为用户没说具体是哪个视频，所以我就不传 `file_path` 参数了。”
4.  **智能体 (`agent.py`)**：接收到LLM的指令，就在后端代码里找到对应的 `helmet_detection_tool`，并执行 `detect_from_file()`。
5.  **工具执行 (`tools.py`)**：你的Python代码开始运行。它发现 `file_path` 参数是空的，于是就自动去 `output_parts` 文件夹里找到了第一个MP4文件进行分析，最后返回一个包含检测结果的JSON对象。
6.  **结果反馈**：智能体将工具的执行结果（那个复杂的JSON）再次发给**千问LLM**，并问：“结果我拿到了，你看看怎么跟用户汇报比较好？”
7.  **千问LLM (大脑)**：分析了那个JSON结果，并为你生成了一段通俗易懂、带有分析和建议的专业报告。
8.  **你 (用户)**：在前端看到了这段专业的分析报告，以及整个过程的“决策轨迹”。

---

## 6. 什么是“决策轨迹”？

为了让你能看懂智能体“思考”和“行动”的过程，我们在前端设计了“决策轨迹”这个功能。它不是LLM真正的内部思考过程（那太复杂了），而是**把整个任务的关键步骤可视化地展示出来**：

1.  **用户输入**：你说了什么。
2.  **模型决策**：LLM（或离线策略）决定要调用哪个工具。
3.  **工具执行**：实际执行了哪个工具，返回了什么结果（JSON内容可展开查看）。
4.  **最终回复**：LLM（或离线策略）生成的最终答复。

这让你能清楚地知道每一步发生了什么，增加了系统的透明度和可信度。

---

## 7. 总结

通过“工具”这种模式，我们成功地让你写的Python功能函数，能够被一个强大的“大脑”（LLM）所理解和调度。你不再是直接调用函数，而是通过自然语言“指挥”一个智能的代理，由它来决定何时、如何调用你的功能，极大地提升了系统的智能化和易用性。
